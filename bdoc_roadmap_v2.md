# План развития формата BDoc и редактора

## Этап 1 — Структура документа: добавление недостающих объектов

Цель этапа — довести модель документа до полноты спецификации v0.1 и закрыть разрывы, которые видны при сравнении с бинарными полиграфическими форматами (IDML, INDD, Scribus SLA).

### 1.1 Templates (Master Pages)
- `MasterPage` — шаблон страницы с фиксированными элементами
- `PageTemplate` — типовые композиции (титульная, глава, обычная страница)
- `MarginModel` — поля страницы
- `GridModel` — колонная сетка и направляющие
- `HeaderFooterRule` — правила колонтитулов и автонумерации страниц
- `BaselineGrid` — типографская базовая сетка
- **Именованные направляющие (Guides) как самостоятельные объекты** — привязка к MasterPage или конкретной странице, цвет, свойство "магнитизма" (Snap to Guides). Без этого верстальщик не сможет точно выравнивать фреймы.

### 1.2 ReadingOrder
- `ReadingOrder` — контейнер порядка чтения на странице/развороте
- `ReadingSegment` — отдельный сегмент порядка чтения (ссылка на объект + позиция)

### 1.3 Недостающие объекты страницы
- `Group` — композиционное объединение объектов
- `TableFrame` — таблица как самостоятельный объект (модель + сериализация)
- `LineObject` — линии, разделители, декоративные линейки
- `ClipObject` — объект отсечения видимой области
- `MaskObject` — объект маски прозрачности
- `ArtifactObject` — служебный, невидимый в основном содержании объект
- **Обтекание текста (Text Wrap):** буквицы, иллюстрации и вставки, которые текст должен обтекать по контуру. В `BdocObject`/`TextFrame` нужно свойство `textWrapMode` (None, BoundingBox, Contour) и сам контур обтекания — без этого текст просто налезает на картинку.

### 1.4 Расширение геометрии
- `Ellipse`, `Polygon` — базовые фигуры
- `BezierPath` — произвольный контур
- `CompoundPath` — составная фигура с внешними/внутренними контурами
- `Transform` — перенос/поворот/масштаб как явная сущность, а не побочное свойство
- **Система единиц измерения (Unit System):** координаты не могут быть "просто double" в пикселях. Нужно жёстко хранить единицы: пункты (pt) для шрифтов, миллиметры (mm) для страниц и вылетов, пайки (pica) для сеток. В `Geometry`/`PageModel` должен быть явный коэффициент пересчёта.

### 1.5 Текстовая модель — расширения
- `InlineObject` — якорный объект внутри текста (привязан к позиции в Span)
- `Footnote` — сноска, связанная со Story
- `Reference` — перекрёстная ссылка / гиперссылка
- Story threading: `nextFrameRef` / `previousFrameRef` на `TextFrame` для перетекания текста между фреймами
- **Флаг переполнения (`isOverset`):** если текст истории длиннее суммарной площади всех связанных фреймов, модель должна явно знать, что часть текста скрыта, а не просто "отрезать молча".

### 1.6 Стили и иерархия объектов (Object Styles & Anchoring)
- `ObjectStyle` — стиль для графических фреймов (обтекающий контур, скругление углов, эффекты прозрачности, привязка к слою по умолчанию) и поле `objectStyleRef` у каждого `BdocObject`. Изменение стиля разом меняет тысячи фреймов по книге.
- **Математика anchored objects:** графический фрейм или таблица могут быть привязаны к конкретному слову в тексте. При редактировании текста привязанный объект должен физически пересчитывать `X`/`Y` относительно новой позиции строки, а не оставаться на месте.

### 1.7 Цвет и Swatches
- `Swatch` — именованный образец цвета (CMYK / RGB / Spot / Lab)
- `ColorProfile` — ICC-профиль, привязываемый к документу или изображению
- Замена цвета-строки в стилях/фигурах на `swatchRef`
- Конвертация CMYK → RGB для экранного превью (рендерер)

### 1.8 Prepress-геометрия страницы
- `bleed`, `trimBox`, `safetyMargin` — поля модели `Page`
- `PrintMarks` — crop marks, registration marks, color bars как графические объекты в bleed-зоне
- `trapping` / `overprint` — флаги на `Fill` / `Stroke`

### 1.9 Дополнительно (из анализа бинарных форматов)
- Hyperlinks как отдельная сущность (не путать с `Reference`)
- Anchored objects — уточнение модели `InlineObject` под привязку к символу
- Document history / undo trail на уровне модели (опционально, если не реализуем на уровне UI)
- **Расширяемость (`customData`):** поле `customData: Map<String, Object>` у `BdocObject` как "песочница" для сторонних плагинов (ИИ-метаданные, AR-маркеры и т.п.) без нарушения валидации ядра.

### Обновления валидатора
- `BdocIntegrityValidator`: проверка новых ссылок — `templateRef`, `marginRef`, `readingOrderRef`, `swatchRef`, `objectStyleRef`, `nextFrameRef`/`previousFrameRef`
- Проверка целостности цепочек threading (нет циклов, нет висячих ссылок)

## Этап 2 — Функциональность программы: превращение редактора в издательскую систему

Цель этапа — добавить в BDoc Editor функциональность, без которой ни одна модель документа не делает программу пригодной для реального издательского процесса.

### 2.1 Master Pages в UI
- Применение шаблона страницы к конкретной странице
- Переключение шаблона
- Override отдельных элементов шаблона на уровне конкретной страницы
- Редактор самого шаблона (поля, сетка, колонтитулы, направляющие)

### 2.2 Preflight-модуль (расширение BdocIntegrityValidator)
- Проверка эффективного DPI изображений после масштабирования
- Проверка цветового режима (CMYK vs RGB) относительно назначения документа
- Проверка bleed/trim на соответствие требованиям печати
- Проверка отсутствующих или не встроенных шрифтов
- Итоговый отчёт preflight с уровнями Error/Warning/Info
- **Проверка прозрачности и эффектов (Transparency Flattening):** старые типографские РИПы ломаются на прозрачных слоях (например, слой сканов-подложек с opacity < 100%). Preflight должен предупреждать о необходимости сведения слоёв перед печатью.

### 2.3 Управление шрифтами
- Панель со списком используемых шрифтов и их статусом (installed/missing/embedded)
- Встраивание (subset embedding) при экспорте

### 2.4 Управление изображениями (Links panel)
- Список linked/embedded ресурсов
- Статусы Missing / Modified
- Обновление (relink) ресурса

### 2.5 Цветоделение и Soft Proofing
- Просмотр отдельных красочных пластин (separations preview)
- Soft proofing с применением ICC-профиля к экранному рендеру

### 2.6 Длинные документы
- Автосборка Table of Contents на основе ролей Paragraph (`title`, `heading`)
- Alphabetic Index
- Footnotes / Endnotes — редактирование и авторазмещение
- Сквозная нумерация страниц/глав

### 2.7 Таблицы — полноценный редактор
- Ввод и редактирование содержимого ячеек
- Объединение/разбиение ячеек
- Table styles (переиспользуемое оформление)

### 2.8 Типографика
- Кернинг и трекинг
- Переносы слов (hyphenation)
- Полная выключка по формату (justify) с учётом пробелов между словами
- Базовые OpenType-фичи (лигатуры)
- **Архитектура разрыва строк (Paragraph Composer):** построчный расчёт (как в Word) даёт "коридоры" пробелов при justify. Нужен многопроходный алгоритм уровня Кнута-Пласса, который оценивает весь абзац целиком и распределяет плотность пробелов равномерно — движок рендеринга текста должен проектироваться под это сразу, а не как доработка позже.

### 2.9 Экспорт и Package
- Экспорт в PDF/X-1a и PDF/X-4
- "Collect for Output" — сбор всех шрифтов и изображений в отдельную папку для передачи в типографию
- Встраивание bleed/trim/print marks в экспортируемый PDF
- **Модуль преобразования цвета (CMS):** при экспорте в PDF/X-1a экранные RGB-изображения обязаны быть переведены в CMYK. Модели мало знать профиль — нужна интеграция с реальной Color Management System (например, LittleCMS), которая физически пересчитывает байты растра из RGB в целевой CMYK-профиль типографии.

### 2.10 Undo/Redo и History
- Полная история операций редактирования (не только последнее действие)
- Отображение истории в UI (опционально)

### 2.11 Batch-обработка
- Применение стиля/шаблона сразу к нескольким страницам/главам
- Простые макросы для повторяющихся операций в сериях документов

### 2.12 Специализированные DTP-инструменты
- **Gap Tool** — изменение зазора между соседними фреймами без изменения их общих внешних границ: левый фрейм сжимается, правый расширяется, сетка сохраняется
- **Scissors Tool / Pen Tool** — интерактивное редактирование узлов `BezierPath` на холсте; разбиение фрейма на два независимых объекта
- **Eyedropper Tool** — мгновенное копирование стилей: считать шрифт/размер/кернинг/цвет с одного текста и применить к другому

## Этап 3 — Оптимизация и расширяемость

### 3.1 Кэш отрисованных страниц (Render Cache)
Текущая модель ленивой загрузки страниц хороша, но векторный текст не должен перерисовываться заново при каждом мелком сдвиге холста. Страница один раз рендерится в растровый буфер (bitmap) в памяти; при скролле процессор просто двигает готовую картинку, пересчитывая вектор только при Resize или изменении текста. Критично для книг на 1000+ страниц на слабом железе.

### 3.2 Архитектура плагинов и SDK (модель IntelliJ Platform)
Чтобы вокруг BDoc выросла экосистема платных и бесплатных расширений, архитектуру SDK нужно закладывать на уровне базовых интерфейсов ядра, а не пристраивать позже.

**Слой Core API (точки расширения в модели):**
- `DocumentLifecycleListener` — события открытия/сохранения/закрытия документа
- `PageLayoutListener` — вызывается при перемещении/ресайзе фрейма, позволяет плагинам ИИ-вёрстки автоматически корректировать макет

**Слой UI API (точки расширения в интерфейсе):**
- Панели инструментов, свойств и меню в `BdocEditorApp` не должны быть жёстко зашиты в коде
- Интерфейс строится динамически из конфигурации плагина (аналог `plugin.xml` в IDEA): плагин регистрирует свой инструмент декларативно, ядро считывает конфиг при старте и само добавляет кнопку на `ToolBar`

**Примеры плагинов для будущего маркетплейса:**
- AI-OCR и реставрация — распознавание грязного скана и автогенерация `TextFrame` поверх картинки на слое background
- Импортёры редких форматов — конвертация из старых версий QuarkXPress, Corel Ventura и т.п. в `.bdoc`
- Автоматическая вёрстка каталогов — подключение к базе данных (1С, Magento) и генерация печатного каталога на сотни страниц по шаблону

### 3.3 Критическое инженерное требование
Чтобы концепция SDK была реализуема, классы в пакете `org.example.bdoc.model` должны оставаться "чистыми" POJO и не знать ничего о JavaFX, `Canvas` или кнопках интерфейса. Управлять моделью через открытый интерфейс API должно быть одинаково возможно как из собственного UI-приложения, так и из любого стороннего плагина. Это разделение стоит закладывать до начала Этапа 1, а не после.

## Порядок выполнения

1. Этап 1 выполняется последовательно по подпунктам 1.1 → 1.9, каждый — с обновлением модели, сериализации (JSON/CBOR) и валидатора.
2. Этап 2 стартует после того, как соответствующий объект модели из Этапа 1 готов (например, Master Pages в UI — только после 1.1; Preflight — после 1.7 и 1.8, так как проверяет цвет и bleed).
3. Экспорт в PDF/X (2.9) — финальная точка, требующая завершения обоих этапов почти целиком.
4. Разделение Model/UI (3.3) закладывается до начала Этапа 1, чтобы не переписывать архитектуру задним числом.
