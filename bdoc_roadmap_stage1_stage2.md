# План развития формата BDoc и редактора — Этап 1 и Этап 2

## Этап 1 — Структура документа: добавление недостающих объектов

Цель этапа — довести модель документа до полноты спецификации v0.1 и закрыть разрывы, которые видны при сравнении с бинарными полиграфическими форматами (IDML, INDD, Scribus SLA).

### 1.1 Templates (Master Pages)
- `MasterPage` — шаблон страницы с фиксированными элементами
- `PageTemplate` — типовые композиции (титульная, глава, обычная страница)
- `MarginModel` — поля страницы
- `GridModel` — колонная сетка и направляющие
- `HeaderFooterRule` — правила колонтитулов и автонумерации страниц
- `BaselineGrid` — типографская базовая сетка

### 1.2 ReadingOrder
- `ReadingOrder` — контейнер порядка чтения на странице/развороте
- `ReadingSegment` — отдельный сегмент порядка чтения (ссылка на объект + позиция)

### 1.3 Недостающие объекты страницы
- `Group` — композиционное объединение объектов
- `TableFrame` — таблица как самостоятельный объект (модель + сериализация)
- `LineObject` — линии, разделители, декоративные линейки
- `ClipObject` — объект отсечения видимой области
- `MaskObject` — объект маски прозрачности
- `ArtifactObject` — служебный, невидимый в основном содержании объект
- **(Дополнение) Сложные вырезы и обтекание (Text Wrap):** Книги часто содержат буквицы (Drop Caps), иллюстрации или вставки, которые текст должен обтекать по контуру. В модель `BdocObject` (или в сам `TextFrame`) необходимо добавить свойство `TextWrapMode` (None, BoundingBox, Contour) и контур обтекания. Без этого текст просто налезет на картинку.

### 1.4 Расширение геометрии
- `Ellipse`, `Polygon` — базовые фигуры
- `BezierPath` — произвольный контур
- `CompoundPath` — составная фигура с внешними/внутренними контурами
- `Transform` — перенос/поворот/масштаб как явная сущность, а не побочное свойство
- **(Дополнение) Система единиц измерения (Unit System):** В DTP-моделях координаты не могут быть «просто double» (пикселями). Документ должен жестко хранить единицы: пункты (pt) для шрифтов, миллиметры (mm) для страниц и вылетов (bdoc_roadm... p. 1), и пайки (pica) для сеток. В `Geometry` или `PageModel` должен быть зашит явный коэффициент пересчета.

### 1.5 Текстовая модель — расширения
- `InlineObject` — якорный объект внутри текста (привязан к позиции в Span)
- `Footnote` — сноска, связанная со Story
- `Reference` — перекрёстная ссылка / гиперссылка
- Story threading: `nextFrameRef` / `previousFrameRef` на `TextFrame` для перетекания текста между фреймами
- **(Уточнение по Story Threading):** Мало добавить `nextFrameRef` (bdoc_roadm... p. 1). Текстовый фрейм должен иметь статус-флаг `isOverset` (выход текста за границы). Если текст истории длиннее, чем физическая площадь всех связанных фреймов, модель должна знать, что часть текста скрыта.

------

### 1.6 Цвет и Swatches
- `Swatch` — именованный образец цвета (CMYK / RGB / Spot / Lab)
- `ColorProfile` — ICC-профиль, привязываемый к документу или изображению
- Замена цвета-строки в стилях/фигурах на `swatchRef`
- Конвертация CMYK → RGB для экранного превью (рендерер)

### 1.7 Prepress-геометрия страницы
- `bleed`, `trimBox`, `safetyMargin` — поля модели `Page`
- `PrintMarks` — crop marks, registration marks, color bars как графические объекты в bleed-зоне
- `trapping` / `overprint` — флаги на `Fill` / `Stroke`

### 1.8 Дополнительно (из анализа бинарных форматов)
- Hyperlinks как отдельная сущность (не путать с `Reference`)
- Anchored objects — уточнение модели `InlineObject` под привязку к символу
- Document history / undo trail на уровне модели (опционально, если не реализуем на уровне UI)

### Обновления валидатора
- `BdocIntegrityValidator`: проверка новых ссылок — `templateRef`, `marginRef`, `readingOrderRef`, `swatchRef`, `nextFrameRef`/`previousFrameRef`
- Проверка целостности цепочек threading (нет циклов, нет висячих ссылок)

## Этап 2 — Функциональность программы: превращение редактора в издательскую систему

Цель этапа — добавить в BDoc Editor функциональность, без которой ни одна модель документа не делает программу пригодной для реального издательского процесса.

### 2.1 Master Pages в UI
- Применение шаблона страницы к конкретной странице
- Переключение шаблона
- Override отдельных элементов шаблона на уровне конкретной страницы
- Редактор самого шаблона (поля, сетка, колонтитулы)

### 2.2 Preflight-модуль (расширение BdocIntegrityValidator)
- Проверка эффективного DPI изображений после масштабирования
- Проверка цветового режима (CMYK vs RGB) относительно назначения документа
- Проверка bleed/trim на соответствие требованиям печати
- Проверка отсутствующих или не встроенных шрифтов
- Итоговый отчёт preflight с уровнями Error/Warning/Info
- **(Дополнение Preflight): Проверка прозрачности и эффектов (Flattening):** Старые типографские машины (РИПы) ломаются, если в PDF уходят прозрачные слои (например, ваш слой сканов-подложек с opacity < 100%) (bdoc_roadm... p. 2). В Preflight нужно добавить проверку на наличие прозрачности в финальных печатных объектах и предупреждение о необходимости сведения слоев (Transparency Flattening)

### 2.3 Управление шрифтами
- Панель со списком используемых шрифтов и их статусом (installed/missing/embedded)
- Встраивание (subset embedding) при экспорте

### 2.4 Управление изображениями (Links panel)
- Список linked/embedded ресурсов
- Статусы Missing / Modified
- Обновление (relink) ресурса

### 2.5 Цветоделение и Soft Proofing
- Просмотр отдельных красочных пластин (separations preview)
- Soft proofing с применением ICC-профиля к экранному рендеру

### 2.6 Длинные документы
- Автосборка Table of Contents на основе ролей Paragraph (`title`, `heading`)
- Alphabetic Index
- Footnotes / Endnotes — редактирование и авторазмещение
- Сквозная нумерация страниц/глав

### 2.7 Таблицы — полноценный редактор
- Ввод и редактирование содержимого ячеек
- Объединение/разбиение ячеек
- Table styles (переиспользуемое оформление)

### 2.8 Типографика
- Кернинг и трекинг
- Переносы слов (hyphenation)
- Полная выключка по формату (justify) с учётом пробелов между словами
- Базовые OpenType-фичи (лигатуры)
-  **(Критическое уточнение по Типографике): Архитектура разрыва строк (Paragraph Composer):** Вы указали `justify` (bdoc_roadm... p. 3). В Word разрыв строк считается поочередно для каждой строки — из-за этого в тексте появляются огромные дыры («коридоры»). InDesign использует многопроходный алгоритм (алгоритм Кнута-Пласса). Он оценивает *весь абзац целиком*, подбирая переносы так, чтобы плотность пробелов была одинаковой. Если вы хотите обойти конкурентов, ваш движок рендеринга текста должен сразу проектироваться под многопроходный расчет абзаца, а не строки.

### 2.9 Экспорт и Package
- Экспорт в PDF/X-1a и PDF/X-4
- "Collect for Output" — сбор всех шрифтов и изображений в отдельную папку для передачи в типографию
- Встраивание bleed/trim/print marks в экспортируемый PDF
- **(Дополнение по Экспорту): Преобразование цвета (Color Conversion Device):** При экспорте в PDF/X-1a формат обязан перевести все экранные RGB-картинки в CMYK (bdoc_roadm... pp. 2-3). Модели мало просто знать профиль (bdoc_roadm... p. 2) — на Этапе 2 движок обязан иметь модуль CMS (Color Management System, например, интеграцию с LittleCMS), чтобы физически пересчитывать массивы байтов растра из RGB в целевой CMYK-профиль типографии.

------

### 2.10 Undo/Redo и History
- Полная история операций редактирования (не только последнее действие)
- Отображение истории в UI (опционально)

### 2.11 Batch-обработка
- Применение стиля/шаблона сразу к нескольким страницам/главам
- Простые макросы для повторяющихся операций в сериях документов

# Этап 3 Оптимизация и Расширения

**Для работы на слабом железе (1000+ страниц):** Ваша текущая модель ленивой загрузки идеальна. Но добавьте в план пункт: *«Кэш отрисованных страниц (Render Cache)»*. Программа не должна рендерить векторный текст заново при каждом мизерном сдвиге холста. Страница должна один раз отрендериться в растровый буфер (Bitmap Image) в памяти, и при скроллинге процессор должен просто двигать готовую картинку, пересчитывая вектор только при Resize или изменении текста.

**Для архитектуры расширений (Plugins):** Чтобы формат BDoc читали сторонние утилиты, в схему JSON/CBOR для `BdocObject` нужно добавить поле `customData: Map<String, Object>`. Это «песочница». Сторонний разработчик сможет зашивать туда свои метаданные (например, разметку для ИИ-переводчика или маркеры дополненной реальности), не ломая валидацию вашей основной программы



## Порядок выполнения

1. Этап 1 выполняется последовательно по подпунктам 1.1 → 1.8, каждый — с обновлением модели, сериализации (JSON/CBOR) и валидатора.
2. Этап 2 стартует после того, как соответствующий объект модели из Этапа 1 готов (например, Master Pages в UI — только после 1.1; Preflight — после 1.6 и 1.7, так как проверяет цвет и bleed).
3. Экспорт в PDF/X (2.9) — финальная точка, требующая завершения обоих этапов почти целиком.



# Примечание



## 1. Что не учтено в структуре документа (Архитектура InDesign)

В InDesign объекты страниц не висят в воздухе, они жестко подчинены иерархии связей. В структуре BDoc v0.1 не хватает трех важнейших полиграфических концепций:

- **Стили объектов (Object Styles):** Вы учли стили текста (Paragraph/Character) (p. 1), но забыли про графику. В InDesign вы можете создать стиль для «Фрейма иллюстрации» (задать ему дефолтный обтекающий контур (p. 1), скругление углов, эффекты прозрачности (p. 2), привязку к определенному слою). Если поменяется стиль — изменятся тысячи фреймов по всей книге. В вашей модели должна появиться сущность `ObjectStyle` и поле `objectStyleRef` у каждого `BdocObject`.
- **Иерархия связанных объектов (Parent-Child Geometry / Anchoring):** В плане упоминаются `Anchored objects` (p. 2), но важно заложить их математику. В InDesign графический фрейм или таблица могут быть привязаны к конкретному слову в тексте. Когда текст редактируется и сдвигается вниз на 5 страниц, привязанная картинка *физически летит вместе с ним*, автоматически пересчитывая свои `X` и `Y` относительно новой строки.
- **Именованные сетки и направляющие на уровне страниц (Liquid Layout / Object Alignment):** В плане есть `GridModel` (p. 1), но в InDesign направляющие (Guides) — это самостоятельные интерактивные объекты структуры документа. Они могут быть привязаны к Мастер-странице (p. 1) или к конкретной странице, иметь цвет и свойство "магнитизма" (Snap to Guides). Без этого верстальщик не сможет выровнять фреймы ровно.

## 2. Какие инструменты еще нужны в UI (Панель инструментов DTP)

Помимо инструментов Выделения (Selection) и Текста (Text), профессиональный издательский процесс требует специфического инструментария:

- **Инструмент «Зазор» (Gap Tool):** Уникальная и невероятно удобная фича InDesign. Она позволяет менять расстояние *между* фреймами (зазор), не меняя их общие внешние границы. Вы двигаете зазор мышкой — левый фрейм сжимается, а правый расширяется, сохраняя полиграфическую сетку.
- **Инструмент «Ножницы» (Scissors Tool) / Перо (Pen):** Поскольку вы планируете добавить `BezierPath` (произвольный контур) (p. 1), верстальщику нужны инструменты для интерактивного редактирования узлов этого контура прямо на холсте. Кликнули «Ножницами» по фрейму — он разбился на два независимых геометрических объекта.
- **Пипетка свойств (Eyedropper Tool):** Инструмент мгновенного копирования стилей. Верстальщик кликает пипеткой на одно слово (считывает шрифт, размер, кернинг, цвет (p. 2)), а затем проводит по другому тексту, мгновенно применяя эти стили.

## 3. Архитектура SDK и монетизация (Пример IntelliJ IDEA)

Ориентация на модель **IntelliJ Platform** — это самое зрелое бизнес-решение для Open Source проекта. Чтобы построить вокруг BDoc живую экосистему платных и бесплатных расширений, архитектуру SDK нужно закладывать на уровне базовых интерфейсов ядра.

В IntelliJIdea всё построено на концепции **компонентов, расширений (Extension Points) и инверсии управления (IoC/DI)**. Для BDoc-Editor (на Java/JavaFX) это означает следующее:

### А. Слой Core API (Точки расширения в Модели)

Ядро программы должно предоставлять интерфейсы для перехвата любых действий с документом. Плагины должны иметь возможность подписываться на события ядра (Application Listeners):

- `DocumentLifecycleListener` (события открытия, сохранения, закрытия книги).
- `PageLayoutListener` (вызывается, когда фрейм двигают или меняют размер, позволяя плагинам ИИ-верстки автоматически корректировать макет).

### Б. Слой UI API (Точки расширения в интерфейсе)

В JavaFX-коде `BdocEditorApp` все панели (инструменты слева, свойства справа, меню сверху) не должны быть жестко зашиты ("захардкожены"). Интерфейс должен строиться динамически на основе конфигурационного файла плагина (аналог `plugin.xml` в IDEA).

- Плагин регистрирует свой инструмент: `<dtpTool id="myAiTranslator" class="com.vendor.AiTool" icon="/icons/ai.png"/>`
- Ядро считывает этот XML/JSON при старте и само добавляет кнопку на `ToolBar`.

### В. Примеры плагинов, на которых разработчики (и вы через комиссию маркетплейса) смогут зарабатывать:

1. **AI-OCR и Реставрация:** Плагин, который подключает внешнюю нейросеть для автоматического распознавания грязного скана и мгновенной генерации структуры `TextFrame` поверх картинки на слое `BACKGROUND`.
2. **Импортеры редких форматов:** Плагины, добавляющие поддержку конвертации из закрытых старых форматов (например, старых версий QuarkXPress или Corel Ventura) в ваш `.bdoc`.
3. **Автоматические верстальщики каталогов:** Плагин, который подключается к базе данных интернет-магазина (1С, Magento), затягивает JSON с товарами и по шаблону автоматически генерирует печатный каталог на 500 страниц за минуту.

## Резюме по плану

Ваш план готов к реализации Этапа 1. Чтобы концепция SDK взлетела в будущем, сейчас вам нужно сделать главное инженерное действие: **полностью изолировать UI от Модели**. Классы в пакете `org.example.bdoc.model` не должны знать ничего о JavaFX, `Canvas` или кнопках. Они должны быть "чистыми" данными (POJO), которыми через открытый интерфейс API сможет управлять как ваше UI-приложение, так и любой сторонний плагин.
